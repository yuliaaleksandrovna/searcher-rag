# Searcher — Поисковик/RAG (Кейс №2)

Поисковый сервис поверх статей Wikipedia: полнотекстовый поиск через Elasticsearch,
опциональная суммаризация результата через LLM (Claude), авторизация с ролями
(reader/writer/owner), аудит поисковых запросов и «избранное» в SQLite.

## Архитектура

- **Elasticsearch** — полнотекстовый индекс статей (`title`, `content`, `url`)
- **SQLite** (`data/searcher.db`) — пользователи, роли, аудит поиска, избранное
- **FastAPI** (`app/`) — REST API + отдаёт статический UI
- **Claude Haiku** (опционально) — суммаризация результатов поиска

Подробнее — см. `docs/ER-diagram.md` (схема БД) и `docs/report.md` (отчёт).

## 1. Установка

```bash
cd "searcher 2"
python3.10 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Откройте `.env` и при желании поменяйте `JWT_SECRET` на случайную строку
(для учебного стенда можно оставить дефолт, но **не для продакшена**).
`ANTHROPIC_API_KEY` — опционально: без него суммаризация работает в
экстрактивном режиме (без LLM).

## 2. Поднять Elasticsearch

```bash
docker-compose up -d
```

Подождите ~20–30 секунд, пока контейнер станет healthy:

```bash
docker-compose ps
```

## 3. Наполнить данные

`data/articles.json` не хранится в репозитории (файл большой и легко
пересоздаётся) — его нужно сгенерировать перед первым запуском:

```bash
python scripts/parser.py
```

Коллекция должна содержать ≥5000 документов. Текст статей парсер берёт из
датасета `wikimedia/wikipedia` на HuggingFace Hub (потоково, без обращений к
MediaWiki API — так и быстрее, и не упирается в анонимный rate-limit
Wikipedia, который при попытке вытянуть текст тысяч статей через
`prop=extracts` быстро выдаёт постоянные 429). После сбора текста скрипт
отдельно дозапрашивает категории статей через MediaWiki API
(`prop=categories`) небольшими батчами — но только для уже отобранных ~6000
статей, а не для всех кандидатов, поэтому укладывается в лимит. Итоговое
число документов и статей с категориями печатается в конце; если документов
меньше 5000, скрипт явно предупредит об этом (увеличьте `TARGET_DOCS` в
скрипте и запустите заново).

Загрузить статьи в Elasticsearch (пересоздаёт индекс `articles` с нуля):

```bash
python scripts/indexer.py
```

Ожидаемый вывод: `Index now contains N documents`, где `N ≥ 5000`.

## 4. Создать первого пользователя-владельца (owner)

Без этого некому будет зайти в `/admin/*`:

```bash
python scripts/seed_owner.py --username admin --password admin123456
```

(без `--username`/`--password` скрипт спросит их интерактивно). Повторный запуск
с тем же логином повышает существующего пользователя до owner и обновляет пароль.

## 5. Запустить сервер

```bash
uvicorn app.main:app --reload
```

- UI: http://localhost:8000
- Swagger-документация API: http://localhost:8000/docs
- Health-check: http://localhost:8000/health

## 6. Проверить, что всё работает

### Автоматический smoke-тест

Прогоняет весь сценарий: регистрация → логин → поиск (с авторизацией и без) →
запрет сохранения для reader → повышение роли owner'ом → сохранение статьи →
доступ к `/admin/*` (запрет для reader, разрешение для owner).

```bash
python scripts/smoke_test.py --owner-username admin --owner-password admin123456
```

Ожидаемый вывод — все строки `[PASS]`, в конце `All checks passed.`.

### Ручная проверка через UI (сценарии для записи демо)

1. Открыть http://localhost:8000 — экран логина.
2. Зарегистрировать нового пользователя (роль по умолчанию — `reader`).
3. Выполнить поиск (например, `machine learning`), включить чекбокс «Суммаризация».
4. Выбрать категорию в выпадающем списке «Категория» и убедиться, что выдача
   сузилась только до статей с этой категорией Wikipedia.
5. Убедиться, что кнопки «Сохранить» под результатами не видно (у reader нет прав).
6. Открыть вкладку «История» — увидеть свой поисковый запрос.
7. Залогиниться как `admin` (owner), перейти во вкладку «Админ»:
   - повысить роль тестового пользователя до `writer`,
   - убедиться, что появилась статистика и общая история всех пользователей.
8. Разлогиниться, зайти под тестовым пользователем (теперь `writer`) — кнопка
   «Сохранить» на карточках результата появилась; сохранить статью и проверить
   вкладку «Избранное».
9. Попробовать зайти во вкладку «Админ» под `writer`/`reader` — вкладка не
   показывается в UI; прямой вызов `GET /admin/users` без токена owner вернёт 403.

## Роли и права

| Роль   | Поиск | Своя история/избранное | Сохранять статьи | Чужие данные | Управление ролями |
|--------|-------|-------------------------|-------------------|--------------|-------------------|
| reader | ✅    | ✅ (только свои)         | ❌                | ❌           | ❌                |
| writer | ✅    | ✅                       | ✅                | ❌           | ❌                |
| owner  | ✅    | ✅                       | ✅                | ✅ (всё)     | ✅                |

Пароли хранятся только в виде bcrypt-хэшей (см. `app/auth.py`), в открытом
виде — никогда.

## Структура проекта

```
app/
  main.py          — FastAPI-приложение, /search, /categories, /auth/*, /me
  db.py            — подключение к SQLite (SQLAlchemy)
  models.py        — таблицы Role/User/SearchLog/SavedArticle
  auth.py          — хэширование паролей, JWT, зависимости FastAPI
  schemas.py       — pydantic-схемы запросов/ответов
  summarizer.py    — суммаризация через Claude / экстрактивный fallback
  routers/
    history.py     — GET /history/me
    saved.py       — POST/GET /saved, DELETE /saved/{id}
    admin.py       — GET /admin/history, /admin/stats, /admin/users, PATCH .../role
scripts/
  parser.py        — сбор статей (текст — из датасета wikimedia/wikipedia, ≥5000 шт.,
                     с дедупликацией; категории — точечно через MediaWiki API)
  indexer.py       — загрузка статей в Elasticsearch
  seed_owner.py    — создание/повышение первого owner-пользователя
  smoke_test.py    — end-to-end проверка сценариев
static/index.html  — UI (логин, поиск, избранное, история, админка)
docs/              — ER-диаграмма, отчёт, шаблон описания вклада участников
```

## Известные ограничения (для отчёта)

- JWT не отзывается до истечения срока действия (нет чёрного списка/refresh-токенов) —
  осознанное упрощение для учебного MVP.
- Гибридный поиск (BM25 + векторный) не реализован — используется только
  полнотекстовый поиск Elasticsearch на основе BM25 (`multi_match` +
  `fuzziness`); BM25 — алгоритм ранжирования Elasticsearch по умолчанию, он
  и закрывает требование «алгоритм поиска» без дополнительного кода.
- Фильтрация выдачи — по одной категории Wikipedia за раз (`?category=...`),
  без мультивыбора и без явного ранжирования сверх штатного BM25-скоринга.
- Дедупликация статей при парсинге — только по точному совпадению хэша
  контента; почти-дублирующиеся, но не идентичные статьи не отсеиваются.
