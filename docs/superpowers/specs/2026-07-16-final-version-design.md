# Финальная версия «Поисковик/RAG» (Кейс №2) — дизайн

Дата: 2026-07-16
Статус: одобрено пользователем, готово к планированию реализации

## Контекст

Проект — поисковый MVP поверх Wikipedia-статей: `scripts/parser.py` тянет статьи,
`scripts/indexer.py` грузит их в Elasticsearch, `app/main.py` отдаёт `/search` с
опциональной суммаризацией через Claude (`app/summarizer.py`), `static/index.html` —
простой UI. Данные (30/30 статей) и UI уже в порядке (предыдущая итерация ревью).

Не хватает для сдачи практики по критериям (слайды 8–13 презентации):
- ролей/авторизации (слайд 11)
- реляционной БД + SQL-запросов с JOIN/GROUP BY/подзапросами (слайд 13)
- ER-диаграммы, отчёта, инструкции запуска, описания вклада участников (слайд 8)
- нескольких мелких багов (deprecated ES API, XSS в href, hardcoded URL, нет .gitignore)

## Архитектура

### Данные: SQLite + SQLAlchemy

Новый файл `data/searcher.db`, отдельный от Elasticsearch (тот остаётся источником
полнотекстового поиска). Таблицы:

```
roles            (id PK, name)                          — reader / writer / owner (сид)
users            (id PK, username UNIQUE, password_hash,
                   role_id FK -> roles.id, created_at)
search_logs      (id PK, user_id FK -> users.id, query,
                   results_count, created_at)             — аудит поисковых запросов
saved_articles   (id PK, user_id FK -> users.id,
                   article_url, article_title, saved_at)  — «избранное»
```

Связи: `roles 1:N users`, `users 1:N search_logs`, `users 1:N saved_articles`.
Пароли — только хэши (`bcrypt`), не открытый текст.

### Матрица прав

| Роль   | Поиск | Своя история/избранное | Сохранять статьи | Чужие данные | Управление ролями |
|--------|-------|------------------------|-------------------|--------------|-------------------|
| reader | ✅    | ✅ (только свои)        | ❌                | ❌           | ❌                |
| writer | ✅    | ✅                       | ✅                | ❌           | ❌                |
| owner  | ✅    | ✅                       | ✅                | ✅ (всё)     | ✅                |

### Авторизация

- JWT (HS256, `PyJWT`), секрет и время жизни токена — из `.env`
  (`JWT_SECRET`, `JWT_EXPIRE_MINUTES`)
- `POST /auth/register` — создаёт пользователя с ролью `reader`, возвращает токен
- `POST /auth/login` — проверяет пароль, возвращает токен
- Зависимость `get_current_user` — достаёт и валидирует Bearer-токен
- Зависимость `require_role(*roles)` — 403, если роль пользователя не подходит

### API (новое и изменённое)

| Метод/путь | Кто | Что делает |
|---|---|---|
| `POST /auth/register` | все | регистрация, роль по умолчанию reader |
| `POST /auth/login` | все | логин, выдача JWT |
| `GET /me` | авторизован | текущий пользователь + роль |
| `GET /search` | авторизован (было публично) | поиск, теперь пишет строку в `search_logs` |
| `GET /history/me` | авторизован | своя история поиска |
| `POST /saved` | writer, owner | сохранить статью в избранное |
| `GET /saved` | авторизован | своё избранное |
| `DELETE /saved/{id}` | владелец записи или owner | удалить из избранного |
| `GET /admin/history` | owner | история всех пользователей (JOIN users+roles+search_logs) |
| `GET /admin/stats` | owner | топ запросов (GROUP BY), пользователи выше среднего по числу поисков (подзапрос) |
| `GET /admin/users` | owner | список пользователей и ролей |
| `PATCH /admin/users/{id}/role` | owner | смена роли пользователя |

### Фронтенд (`static/index.html`)

- Экран логина/регистрации перед поиском; токен — в `localStorage`
- Шапка: `username (role)` + Logout
- Кнопка «Сохранить» на карточке результата — видна только writer/owner
- Простая админ-панель (таблица пользователей + смена роли, вся история) — видна только owner
- XSS-фикс: `escHtml(r.url)` при подстановке в `href`

### Багфиксы

- `scripts/indexer.py`: читать `ELASTICSEARCH_URL` из `.env` (было захардкожено)
- `app/main.py`: `es.search(index=INDEX, query=..., highlight=..., size=...)` вместо
  deprecated `body=`
- `.gitignore`: `.venv/`, `__pycache__/`, `.env`, `*.db`

### Тестирование

- `scripts/smoke_test.py` — скрипт на `requests`, прогоняет сценарий: register → login
  → search (пишется лог) → сохранить статью → попытка admin-доступа читателем
  отклонена (403) → admin видит чужую историю. Пользователь запускает сам и смотрит
  на результат в консоли (короткий, не долгий процесс — можно и через агента, но по
  договорённости все точки входа запускает сам пользователь для наглядности логов).
- Ручная проверка через UI: обе роли, попытка reader'а увидеть admin-панель (не должна
  быть доступна ни в UI, ни через прямой вызов API).

## Границы (что не делаем)

- Не трогаем `app/summarizer.py` и `scripts/parser.py` — они уже рабочие
- Не добавляем refresh-токены/logout-blacklist — JWT просто истекает по времени, для
  учебного MVP этого достаточно
- Не делаем N:N связей искусственно — все реальные связи здесь 1:N, чего достаточно
  по критериям (слайд 9: "1:1, 1:N, N:N → разрешены", не "обязательны все три")
